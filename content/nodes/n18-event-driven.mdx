<HandsOnHook scene="忙等 vs 优雅">

N17 你写了两种监听按钮的代码：

```python
# 方式 A：忙等
while True:
    if button.is_pressed:
        play_sound()
    time.sleep(0.05)

# 方式 B：事件
button.when_pressed = play_sound
pause()
```

**两段代码都"工作"，但完全不同的编程范式。** 方式 A 让 Pi 一直转圈检查（耗 CPU）；方式 B 让 Pi 闲着，事件触发时再被唤醒。

20 分钟读完你会明白：**事件驱动**是嵌入式 + Web 后端 + GUI 应用的统治范式。学会这个心智模型，你能读懂 90% 的现代代码。

</HandsOnHook>

### 轮询（Polling）—— 主动查

```python
import time
from gpiozero import Button

button = Button(17)

while True:
    if button.is_pressed:
        print("按了")
    time.sleep(0.05)         # 必须 sleep，否则 CPU 爆
```

**特征：**
- 你的主循环掌控节奏
- 简单直接，代码线性好读
- 缺点：CPU 一直转，错过短按，多事件混着难管理

### 事件驱动（Event-driven）—— 被动响应

```python
from gpiozero import Button
from signal import pause

button = Button(17)
button.when_pressed = lambda: print("按了")
pause()                      # 永远阻塞 — 让事件循环活着
```

**特征：**
- 你只声明"什么事件 → 触发什么动作"
- 程序大部分时间在睡眠，事件来了硬件中断唤醒
- CPU ≈ 0%，多事件互不干扰
- 缺点：代码"跳来跳去"难一眼读懂

<Misconception
  myth="事件驱动比轮询'更新更高级' — 应该全用事件驱动"
  truth="两种范式各有适用场景。事件驱动适合「等输入」(按钮/网络/UI)；轮询适合「定时检查」(每分钟读传感器)。混着用最常见。"
  why="发声盒的按钮 = 事件（不知道啥时候按）。温度计每分钟读一次 = 轮询（你掌控节奏）。把传感器轮询硬改成事件 = 复杂化；把按钮硬改成轮询 = 浪费 CPU。选错范式 = 项目变难。"
/>

### `pause()` 在做什么

```python
from signal import pause
pause()
```

这一行让程序"永远暂停在这里"。背后实际是：

1. Python 进程进入 `pause()` 系统调用
2. 操作系统挂起进程（不占 CPU）
3. 硬件中断（按钮按下）触发 → OS 唤醒进程 → 进程跑回调函数 → 函数返回 → 进程再进入暂停

**没有 `pause()` 程序立刻退出**，事件回调来不及触发。这是事件驱动代码的"心脏"。

<PredictThenReveal
  question="我代码末尾没写 pause()，只有 button.when_pressed = on_press。然后 python3 main.py 跑起来 —— 按按钮会触发 on_press 吗？"
  hint="想想'程序退出'跟'事件等待'的关系。"
  answer="不会。脚本跑到末尾立刻退出，进程结束，所有事件订阅都没了。即使你飞快地按按钮 —— 中断到达时进程已经不存在。pause() 或者 while True: time.sleep(1) 这种'让进程活下去'的语句是事件驱动的必需品。"

### Lambda：一行函数

```python
# 等价的两种写法

# 写法 1：显式 def
def on_press():
    print("按了")
button.when_pressed = on_press

# 写法 2：lambda（匿名函数）
button.when_pressed = lambda: print("按了")
```

`lambda:` = "我要定义一个没名字的函数"。后面跟函数体。Python 里 lambda 限制：只能一个表达式（不能多行）。简单回调用 lambda 干净。

### 完整发声盒骨架（N19 之前的样子）

```python
from gpiozero import Button
from signal import pause
from pathlib import Path
import random

# 配置
BUTTON_PIN = 17
SOUND_DIR = Path(__file__).parent / "sounds"
SOUNDS = list(SOUND_DIR.glob("*.mp3"))

def play_random_sound():
    """按下按钮时调用 — N19 会把 print 替换成真播放"""
    chosen = random.choice(SOUNDS)
    print(f"播放：{chosen.name}")
    # 这里 N19 会加 pygame.mixer.music.load(chosen).play()

def setup():
    button = Button(BUTTON_PIN, bounce_time=0.1)
    button.when_pressed = play_random_sound
    return button

if __name__ == "__main__":
    button = setup()
    print(f"发声盒就绪 — {len(SOUNDS)} 个音频")
    pause()                  # 永远响应按钮，直到 Ctrl+C
```

**整个发声盒不到 20 行，因为事件驱动让你只写"业务逻辑"，所有"等待 + 调度"由 gpiozero + OS 处理。**

## Try · 动手验证你真懂了

**真上手对比** 两种范式（要 N16 + N17 电路）：

### 实验 1：轮询版按钮控 LED

```python
# poll.py
from gpiozero import LED, Button
import time

led = LED(17)
button = Button(18)

while True:
    led.value = button.is_pressed   # 0/1 直接互转
    time.sleep(0.05)
```

跑起来：按按钮 LED 亮，松开灭。**但 Pi CPU 用 top 看应该有几个百分点**。

### 实验 2：事件版（同样效果）

```python
# event.py
from gpiozero import LED, Button
from signal import pause

led = LED(17)
button = Button(18)

button.when_pressed = led.on
button.when_released = led.off

pause()
```

跑起来：按按钮 LED 亮，松开灭。**Pi CPU 几乎 0%**。

**两段代码同样效果，CPU 占用差几十倍。** 这就是事件驱动的威力。
